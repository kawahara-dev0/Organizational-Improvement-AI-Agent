# 組織改善 AI エージェント — 理解に基づく設計書（要約版）

本文書は `design.md` の仕様を読み取り、実装・レビュー時に参照しやすいよう再構成したものです。原文と矛盾する記述は意図していません。

---

## 1. システムの目的と価値

| 観点 | 内容 |
|------|------|
| 目的 | 従業員の主観的な悩み・不満と、客観的な社内データ（方針・財務など）を橋渡しし、**構造的な改善提案**につなげるプロフェッショナル向けプラットフォーム。RAG で社内ナレッジを参照する。 |
| 匿名性 | 従業員を保護しつつ、**PII に該当しないメタデータ**で組織の健康状態を把握する。 |
| AI | **モデルルーター**で、論理分析（Claude）と高速処理（Gemini）をタスクの複雑さに応じて切り替える。 |
| 成果物 | 感情的な訴えを、**優先度付きの経営向け提案**に再構成する。 |

---

## 2. アーキテクチャと技術スタック

| 層 | 技術 |
|----|------|
| フロントエンド | Next.js 15（App Router）、Tailwind CSS（`package.json`＋ロックファイルでバージョン固定を推奨） |
| バックエンド | Python 3.12、FastAPI |
| データベース | Supabase（PostgreSQL + pgvector） |
| オーケストレーション | LangChain / LangGraph |
| AI モデル | **Claude 4.6 Sonnet**（深い推論・正式提案生成など）、**Gemini 2.0 Flash-Lite**（ルーティング・RAG 全般のデフォルト） |

**開発・検証時の方針**: API コスト削減のため、**機能はすべて Gemini 2.0 Flash-Lite で動かす**。Claude へのルーティング**ロジックは実装するが**、本番相当の切り替えは**環境変数**でのみ有効化する想定。

---

## 3. アクター

- **User（従業員）**: 個人的な相談や組織への提言を行う。
- **Manager（管理者）**: 組織の健全性の把握、提案のレビュー、組織コンテキスト（方針・体制など）の更新。

---

## 4. ユースケースと処理フロー

### UC-1: AI による従業員相談（誰でも）

**ゴール**: 従業員に即時価値を与えつつ、匿名化されたデータで組織マッピングに使う。

1. **入力**  
   - 任意で `#department` を選択（選択肢は UC-Admin で管理）。  
   - チャットで状況を記述。

2. **リアルタイム分類・ログ（Gemini 2.0 Flash-Lite）**  
   会話から以下を抽出し、**会話の進行に応じて DB を継続更新**する。  
   - `consultations.department`  
   - `consultations.category`（例: 報酬、人間関係）  
   - `consultations.severity`（0〜5。抽象的・未確定は 0）

3. **文脈分析（LLM + RAG）**  
   - `#knowledge_base` を検索し、社内方針・戦略文書を参照。  
   - 回答は **個人向けアドバイス**（実践的・共感的）と **構造的視点**（ナレッジと組織フレームに基づく根本原因）に分類。

4. **モデルルーティング（既定: すべて Gemini）**  
   次の**いずれか**を満たすとき **Claude 4.6 Sonnet** に切り替える設計（本番では env で有効化）。  
   - **高リスク**: `severity` が **4 以上**  
   - **正式提案生成**: `is_submitted` が True で **`proposal` を生成する処理**  
   - **高度な戦略分析**: Top-down モードでの複雑な仮説検証・政策ドラフト（分析クエリ）

5. **フィードバック**  
   - アドバイスへの Like/Dislike → `consultations.feedback` に保存。

---

### UC-2: 正式提案へのエスカレーション（提出フェーズ）

**前提**: UC-1 の文脈を、正式な記録に変換する。

1. **開始**: 「マネージャーに提出」などで正式化。

2. **対話的レビューとプロフェッショナルな言い換え（Claude が想定、開発中は Gemini）**  
   - セッションを **要約（summary）と構造提案（proposal）のドラフト**に統合。  
   - **匿名化**: 固有名詞・PII を検出し、一般化・編集。  
   - **リフレーミング**: 攻撃的・感情的な表現を、**本質的な論点を保った**建設的・客観的なビジネス表現へ。

3. **本人特定の任意項目**  
   - UI で **氏名・メール**を任意入力。  
   - 未入力なら匿名のまま。入力時はマネージャーがヒアリングや表彰の連絡に利用しうる。

4. **確定時の原子更新（単一トランザクション）**  
   「確定して送信」で **1 回の DB 更新**で以下をまとめて行う。  
   - `summary`, `proposal` の確定値  
   - `user_name`, `user_email`  
   - `is_submitted` = True  
   - `admin_status` = `"New"`

---

### UC-Admin: マネージャーダッシュボードと戦略制御

1. **ナレッジベース（RAG コンテキスト）**  
   - PDF / Excel / Docx 等をアップロード・同期。  
   - 自動チャンク化・埋め込みで `#knowledge_base` を更新。  
   - チャンクの CRUD で精度・コンプライアンスを維持。

2. **部門マスタ**  
   - 有効な `department` の一覧を定義 → UC-1 のドロップダウンに反映。

3. **トレンドダッシュボード**  
   - **ヒートマップ**: `category` × `severity` の 2 次元で緊張の偏りを可視化。  
   - **エグゼクティブサマリー**: Gemini が最近のメタデータから箇条書きトレンドを生成

4. **提案レビュー**  
   - `is_submitted` = True のみ一覧。  
   - **Top-down モード**: 特定提案の分析や複数提出の横断で全社方針ドラフト（Claude が想定、**ビルド・テスト中は Gemini**）。  
   - `admin_status` の更新で対応状況を管理。

## 5. データモデル（原文どおりの整理）

### `knowledge_base`

| カラム | 型 | 説明 |
|--------|-----|------|
| id | UUID | PK |
| content | text | ドキュメントから分割したチャンク本文 |
| embedding | VECTOR | 例: Gemini Embedding 768 次元 |
| metadata | JSONB | 元ファイル名、ページ、カテゴリ等 |

### `departments`

| カラム | 型 | 説明 |
|--------|-----|------|
| id | UUID | PK |
| name | text | 部門名 |
| created_at | timestamp | 作成時刻 |

### `consultations`

| カラム | 型 | 説明 |
|--------|-----|------|
| id | UUID | PK |
| department | text | 選択部門 |
| category | text | AI 自動抽出 |
| severity | int | 0〜5（0 は未確定・抽象的） |
| feedback | int | 1=Like, -1=Dislike, 0=未回答 |
| is_submitted | bool | デフォルト False、提出で True |
| summary | text | 提出時に AI が生成 |
| proposal | text | 提出時に AI が生成 |
| user_name | text | 任意 |
| user_email | text | 任意 |
| admin_status | text | `"New"`（初期）, `"In Progress"`, `"Resolved"`, `"Archived"` 等 |
| created_at | timestamp | 作成時刻 |

---

## 6. 実装上の留意点（仕様から読み取れるもの）

- **動的メタデータ**: `department` / `category` / `severity` は会話とともに**最新状態へ更新**される。  
- **二段階の価値**: チャットは即時支援、提出後は組織への正式インプット。  
- **コスト・本番切り替え**: Claude ルートは**実装必須だが、開発では Gemini のみ**が運用ルール。  
- **提出トランザクション**: UC-2 の確定は**一括更新**で不整合を防ぐ。
